# Feature Flags: Tier Cascade Rule\n\n## Problem\n\nBefore this fix, feature flags allowed **independent per-plan settings** without enforcing business logic. This created inconsistencies:\n\n- Admin set \"Daily Edition\" to FREE but left VIP as NULL\n- VIP users would see \"Couldn't load\" errors\n- No automatic inheritance from lower tiers to higher tiers\n\n**Example:** `anytime.sync@gmail.com` (VIP) couldn't access Daily Edition or Morning Brief even though FREE tier had access.\n\n## Solution: Tier Cascade Rule\n\n**If a lower tier is granted access, all higher tiers automatically get access.**\n\n### Tier Hierarchy\n```\nFREE (lowest) → PLUS → PRO → VIP (highest)\n```\n\n### Rules\n\n| When | Then | Exception |\n|------|------|----------|\n| FREE is **ON** | PLUS, PRO, VIP all **ON** | Explicit FALSE overrides |\n| PLUS is **ON** (FREE off/null) | PRO, VIP all **ON** | Explicit FALSE overrides |\n| PRO is **ON** (FREE/PLUS off/null) | VIP **ON** | Explicit FALSE overrides |\n| VIP is **ON** (others off) | Only VIP has access | — |\n\n### Example Scenarios\n\n#### Scenario 1: Enable for all users (FREE ON)\n```json\n{\n  \"feature_id\": \"ai_daily_edition\",\n  \"enabled_free\": true,\n  \"enabled_plus\": null,\n  \"enabled_pro\": null,\n  \"enabled_vip\": null\n}\n```\n**Result:** All tiers (FREE, PLUS, PRO, VIP) have access ✅\n\n#### Scenario 2: Enable for PLUS and up (FREE OFF, PLUS ON)\n```json\n{\n  \"feature_id\": \"ai_plan_my_day\",\n  \"enabled_free\": false,\n  \"enabled_plus\": true,\n  \"enabled_pro\": null,\n  \"enabled_vip\": null\n}\n```\n**Result:** FREE doesn't have access, but PLUS, PRO, VIP all have access ✅\n\n#### Scenario 3: Pro-only (PLUS explicitly OFF despite FREE)\n```json\n{\n  \"feature_id\": \"ai_smart_eisenhower\",\n  \"enabled_free\": true,\n  \"enabled_plus\": false,\n  \"enabled_pro\": true,\n  \"enabled_vip\": null\n}\n```\n**Result:**\n- FREE: ON ✅\n- PLUS: OFF ❌ (explicit false overrides cascade)\n- PRO: ON ✅ (cascades from true)\n- VIP: ON ✅ (cascades from pro)\n\n#### Scenario 4: VIP-only (exclusive feature)\n```json\n{\n  \"feature_id\": \"ai_white_glove\",\n  \"enabled_free\": false,\n  \"enabled_plus\": false,\n  \"enabled_pro\": false,\n  \"enabled_vip\": true\n}\n```\n**Result:** Only VIP has access ✅\n\n## Implementation\n\n### Server-Side: `src/lib/feature-flags.ts`\nThe `isPlanEnabledByOverride()` function applies the cascade rule:\n```typescript\nexport function isPlanEnabledByOverride(\n  userPlan: Plan,\n  flag: FeatureFlagRow | undefined | null\n): boolean | null {\n  // Apply tier cascade: if lower tier is ON, higher tiers inherit\n  const shouldCascade = (targetPlan: Plan): boolean => {\n    if (targetPlan === \"plus\" || targetPlan === \"pro\" || targetPlan === \"vip\") {\n      if (flag.enabled_free === true) return true; // FREE enabled → cascade to all\n    }\n    if (targetPlan === \"pro\" || targetPlan === \"vip\") {\n      if (flag.enabled_plus === true) return true; // PLUS enabled → cascade to PRO, VIP\n    }\n    if (targetPlan === \"vip\") {\n      if (flag.enabled_pro === true) return true; // PRO enabled → cascade to VIP\n    }\n    return false;\n  };\n  // ... resolve value with cascade applied\n}\n```\n\n### Admin API: `src/app/api/admin/feature-flags/route.ts`\nThe `applyCascadeRule()` function normalizes saves:\n```typescript\nfunction applyCascadeRule(\n  free: boolean | null,\n  plus: boolean | null,\n  pro: boolean | null,\n  vip: boolean | null\n): { enabled_free, enabled_plus, enabled_pro, enabled_vip } {\n  // If FREE is ON, enable PLUS/PRO/VIP unless explicitly FALSE\n  if (free === true) {\n    if (plus !== false) cascaded_plus = true;\n    if (pro !== false) cascaded_pro = true;\n    if (vip !== false) cascaded_vip = true;\n  }\n  // ... etc for PLUS and PRO\n  return { enabled_free, enabled_plus, enabled_pro, enabled_vip };\n}\n```\n\n### Client-Side: `src/hooks/use-feature-access.ts` + `src/components/app/feature-matrix.tsx`\nBoth use the same `pickPlanCol()` cascade logic to ensure consistency.\n\n## Testing\n\nRun tests to verify cascade rule:\n```bash\nnpm test -- src/lib/__tests__/feature-flags.cascade.test.ts\n```\n\nTest cases cover:\n- ✅ FREE ON cascades to all\n- ✅ PLUS ON cascades to PRO, VIP\n- ✅ PRO ON cascades to VIP\n- ✅ VIP-only access\n- ✅ Explicit FALSE overrides cascade\n- ✅ All off / all null scenarios\n\n## Fixing the Original Issue\n\nTo fix `anytime.sync@gmail.com` (VIP) Daily Edition access:\n\n1. Go to `/admin/feature-flags`\n2. Find \"Daily Edition\" row\n3. Set FREE to \"On\" (or leave existing)\n4. Leave PLUS, PRO, VIP as unset (null) or \"On\"\n5. Click save\n6. The cascade rule will automatically ensure VIP gets access ✅\n\nNo need to manually enable each tier.\n\n## Migration Guide (if you have existing overrides)\n\nIf you have feature flags set up before this change:\n\n1. The cascade rule is applied **on read** (in `isPlanEnabledByOverride()`)\n2. The cascade rule is applied **on write** (in the admin API)\n3. **No data migration required** — existing flags will be normalized when edited\n4. PRO and VIP will cascade from PLUS.\n\n**Q: Can I disable a feature for a specific tier?**\n\nA: Yes! Set that tier to `false`. Example:\n```json\n{\n  \"enabled_free\": true,\n  \"enabled_plus\": false,  // Explicitly OFF despite FREE ON\n  \"enabled_pro\": null,\n  \"enabled_vip\": null\n}\n```\nResult: FREE ✅, PLUS ❌, PRO ✅ (cascades), VIP ✅ (cascades)\n\n**Q: What if I set all to NULL?**\n\nA: The system falls back to `override_plan` or static `minPlan` from code. Cascade doesn't apply when there's no explicit setting.\n\n**Q: Does this affect the database schema?**\n\nA: No. The cascade rule is applied in code (on read) and on admin saves. No schema changes needed.\n"
